# Codex 実装安全基準

このセクションは、このリポジトリで実装を担当する Codex 向けルールの正です。
`CLAUDE.md` や `docs/codex-operations.md` と矛盾する場合は本セクションを優先し、
矛盾を人へ報告して作業を止めてください。

## 1. 役割とスコープ

- Codex は **実装・テスト作成・バグ修正・PR 作成**を担当します。
- 作業は **Claude の plan（ゴール・非ゴール・受け入れ条件・タスク分解）が人に承認された後だけ**始めます。
- 設計変更、受け入れ条件の変更、plan 外へのスコープ拡大は行いません。
- 指示の不足・矛盾・安全上の疑問がある場合は推測で進めず、人へ確認します。

## 2. 絶対に禁止する操作

- `main` への直接 commit、push、force-push。
- Codex 自身による PR の merge、release、tag 作成。
- `OPENAI_API_KEY` の要求、参照、設定、使用、リポジトリへの追加。
- 本番の秘密情報・顧客データを使った開発やテスト。
- 実データを含むログ、画面、設定、エクスポートの Issue / PR への転載。

## 3. 人の明示的承認が必要な操作

次の操作は、対象と目的について人の明示的承認がない限り実行しません。

- Cloudflare への deploy と、`pnpm deploy:*`、`wrangler deploy`。
- GitHub Actions の `workflow_dispatch` 実行。
- remote D1 への query、migration、書き込み、削除。
- LINE API による実送信。テスト送信も含みます。
- 本番・ステージング環境への接続や設定変更。
- 有料 API、課金可能な外部サービス、新規依存関係の追加。

承認がない場合は、mock、fixture、ローカル D1 などで検証します。
人の承認があっても、次項の配信禁止時間帯を上書きすることはできません。

## 4. 配信安全

- 配信禁止時間帯は **`Asia/Tokyo` 基準の 21:00 以上、翌 8:00 未満**です。
- broadcast、multicast、push、シナリオ、リマインダー、予約通知、automation、
  テスト送信など、利用者への能動的な LINE 送信を禁止時間帯の対象とします。
- 受信イベントと同じ処理内で reply token を使う即時 reply だけを例外とします。
- それ以外の例外は設けません。
- 予約時刻は `packages/db/src/scenario-schedule.ts` の `clampToDeliveryWindow()`
  または同等の共通ガードを通します。
- 新規・変更する送信経路では、予約時と実送信直前のどこで禁止帯を保証するかを明示し、
  21:00、23:59、0:00、7:59、8:00 の境界テストを追加します。
- 二重送信・誤アカウント送信・大量誤送信を生む可能性があれば作業を止めます。

## 5. 秘密情報保護

- トークン、チャネルシークレット、API キー、アカウント ID、D1 ID、webhook URL、
  友だち ID、顧客データ、本番エクスポートをコード、コミット、ログ、Issue、PR に含めません。
- `.env.example` 以外の秘密情報を含み得る環境ファイルを表示・転載しません。
- 環境変数や認証情報を一覧出力しません。
- ログ、スクリーンショット、再現手順は必ずマスクします。
- 秘密情報を発見した場合は内容を転載せず、場所と種類だけを人へ報告して作業を止めます。

## 6. 実装開始前チェックリスト

以下をすべて確認してから実装します。

- 対象 Issue と、人が承認した Claude plan がある。
- ゴール、非ゴール、受け入れ条件、変更範囲が明確である。
- 現在のブランチが `main` ではなく、作業ツリーの既存変更を把握している。
- 配信、認証、顧客データ、D1、deploy への影響を確認した。
- 外部接続や課金可能な操作の要否と承認状況を確認した。
- 必要なテスト、完了条件、rollback 方法が明確である。

不足があれば実装を始めず、Issue または PR で確認します。

## 7. 実装と検証

- 受け入れ条件を満たす最小の差分にします。
- 振る舞いを変更する場合は vitest を追加・更新します。
- 外部送信や本番接続は mock 化します。
- 変更対象に応じて `typecheck`、`test`、`build` を実行します。
- docs のみの変更などテスト不要の場合は、未実施理由を PR へ記載します。
- 失敗した検証を隠さず、実行コマンドと結果を PR へ記載します。

## 8. PR と完了条件

- 必ず作業ブランチから PR を作り、対象 Issue を `Fixes #NN` などで紐付けます。
- `.github/PULL_REQUEST_TEMPLATE.md` をすべて埋めます。
- Security Impact、配信影響、未検証項目、rollback 方法を明記します。
- 差分に秘密情報や無関係な変更がないことを確認します。
- Codex は merge しません。
- CI、Claude レビュー、人の最終確認が完了して初めて完了とします。

補足の操作手順は `docs/codex-operations.md` を参照してください。
