> 心電図CRM → be Navigator全体CRM（学び・文房具・イベント・コミュニティ）への設計確定版。
> 状態：**設計完成・未実装**。実装は本設計の合意後に着手。

## 設計の芯（不変の原則）
- **入口は1つのHub**（LIFF `?page=hub`）。登録直後に診断へ直行せず、まず「目的」を1つ選ぶ。
- **状態は「タグ」、値は「metadata」、動きは「シナリオ」**。この3層分離を10,000人規模まで貫く。
- **ルーティングは tag_added**（タグを付ける＝対応シナリオが自動で始まる）。新規ロジックを増やさない。
- **既存V1.0（心電図ルート）は無改修で1ルートとして存続**。
- ブランドは「航海／船長ベッカミ／仲間／全員最初は読めなかった」。やさしく安心・包摂的。

# ① タグ命名（最終版）

## 命名規則
- 形式：**`<カテゴリ1文字>:<日本語の値>`**（例 `R:心電図`、`M:有料会員`）。
- カテゴリ文字：**R**=ルート / **I**=興味 / **M**=会員 / **E**=イベント / **B**=行動 / **D**=診断 / **S**=学習ステータス。
- **内部参照はタグID（UUID・不変）**。プレフィックスと日本語名は人間・管理画面用の表示規約。**値スラッグはいったん公開したら改名しない**（broadcast条件・trigger_tag_idが参照するため）。
- 管理画面ではプレフィックスで自然にグルーピング。カテゴリごとに色を固定（下表）。
- タグは「**真偽の集合所属**」を表す（付いている／いない）。連続値・履歴は metadata で持つ（②⑤参照）。

## カテゴリ別タグ一覧

### ルートタグ R:（オンボーディング起動トリガー＝tag_added の trigger_tag_id）
`R:心電図` `R:循環器` `R:デバイス` `R:心不全` `R:AI` `R:文房具` `R:イベント`
- 役割：付与で対応ルートシナリオが自動 enroll。オンボ完了後は外す or `S:` で完了管理。
- 色：青系。

### 興味タグ I:（恒久の興味プロファイル・セグメント配信用・複数可・基本消さない）
`I:心電図` `I:循環器` `I:デバイス` `I:心不全` `I:AI` `I:文房具` `I:イベント`
- 役割：その人が「興味あり」と表明した記録。ターゲティング配信の母集団。Hub選択や後からの追加で増える。
- ルートとの違い：**R=今走らせる導線（一時的トリガー）、I=長期の興味属性（永続）**。Hub選択時は両方付与。
- 色：水色系。

### 会員タグ M:（課金・在籍ステータス・原則1人1つ）
`M:無料` `M:有料会員` `M:休会` `M:退会` `M:再入会候補`
- 役割：会員ライフサイクル。決済連携（Stripe/Paid Memberships Pro）で自動遷移を将来実装。
- 遷移例：無料→（入会）→有料会員→（解約）→休会/退会→（声かけ対象）→再入会候補。
- 色：緑系。

### イベントタグ E:（イベント単位・動的に増える）
`E:ナースまつり2026` `E:◯◯セミナー2026-07` …（開催ごとに作成）
- 役割：申込・来場の記録、当日リマインド／後追い配信の母集団。
- 命名：`E:<イベント名><年/月>`。終了後も履歴として残す。
- 色：橙系。

### 行動タグ B:（重要アクションの達成フラグ）
`B:診断完了` `B:アプリ登録` `B:グッズ購入` `B:勉強会参加` `B:アーカイブ視聴` `B:入会ページ閲覧` `B:Discord参加`
- 役割：ファネル計測・条件分岐・後追い（例「入会ページ閲覧したが未入会」に再アプローチ）。
- 付与元：tracked-link クリック、フォーム送信、診断完了、将来の決済/参加連携。
- 色：紫系。

### 診断タグ D:（心電図学習タイプ・心電図ルート専用）
`D:F` `D:P` `D:V` `D:L`
- 役割：心電図ルートのタイプ別出し分けの「タグ版」（現状は metadata.diagnosis_type で出し分け。タグでも持つとセグメント配信が容易）。
- 補足：現行は metadata 主体。Dタグは「タイプ別一斉配信」をしたくなった時の母集団用（任意・V1.2で追加可）。
- 色：ピンク系。

### 学習ステータスタグ S:（進捗・状態）
`S:オンボ中` `S:オンボ完了` `S:アクティブ` `S:離脱` `S:休眠`
- 役割：ライフサイクルの段階管理。再エンゲージ配信の母集団。
- 遷移例：オンボ中→オンボ完了→アクティブ→（一定期間無反応）→休眠/離脱。
- 色：グレー系。

> **タグ総数の目安**：R7＋I7＋M5＋B7＋D4＋S5＝35＋イベント(増分)。**カテゴリ制で増えても管理可能**。

# ③ 7ルート設計

> 共通アーク（心電図ルートで実証済）：Day0 歓迎・方向づけ → Day1 場と仲間 → Day2 想い／価値 → Day3 ミニ体験 → Day5 メンバー限定の予告 → Day6 変化・声 → Day7 次の一歩。
> ルートにより「7日教育型」か「情報フィード型（軽量）」かが異なる。最終ゴールは原則**有料会員化**だが、ルートにより一次ゴールが違う。

## 1. 心電図ルート（ecg）＝既存V1.0・無改修
- 目的：心電図を「読める→使える→語れる」へ。看板ルート。
- 対象：心電図に苦手意識／検定志望／現場で使いたいナース。
- Day0〜7：Day0 診断誘導 → 診断結果Push（F/P/V/L）→ Day1 場と仲間（診断接続）→ Day2 ベッカミの想い → Day3 AFクイズ＋タイプ別 → Day5 クルー限定の予告＋タイプ別 → Day6 仲間の声 → Day7 入会案内（直接チェックアウト）。
- 最終ゴール：有料会員化。
- 他ルート導線：Day中に「循環器/デバイス/心不全/AIも学べる」を一言、Day7後に興味追加（Hub再訪）。

## 2. 循環器ルート（circ）＝7日教育型
- 目的：循環器の基礎を体系的に。心電図の土台・周辺知識。
- 対象：循環器初心者、配属直後、基礎を固めたい人。
- Day0 歓迎・全体像 → Day1 場と仲間 → Day2 なぜ体系学習が効くか → Day3 ミニ解説（例：刺激伝導系を1枚で）→ Day5 勉強会・アーカイブ予告（クルー限定）→ Day6 変化の声 → Day7 入会案内。
- 最終ゴール：有料会員化（循環器シリーズ視聴）。
- 他ルート導線：心電図（実地読影へ）、デバイス／心不全（疾患深掘り）。

## 3. デバイス／ペースメーカールート（device）＝7日教育型
- 目的：ペースメーカー・ICD等デバイスの基礎〜判読。
- 対象：CCU/循環器病棟、デバイス外来、苦手意識のある人。
- Day0 歓迎 → Day1 場と仲間 → Day2 デバイスは怖くない（想い）→ Day3 ミニ体験（ペーシング波形の見方1点）→ Day5 デバイス勉強会（外部講師せこさん・月2）の予告 → Day6 声 → Day7 入会案内。
- 最終ゴール：有料会員化（デバイス勉強会参加）。
- 他ルート導線：心電図（波形基礎）、心不全（CRT等で接続）。

## 4. 心不全ルート（hf）＝7日教育型
- 目的：心不全の病態〜治療を現場目線でつなげる。
- 対象：循環器・救急・一般病棟で心不全を看るナース。
- Day0 歓迎 → Day1 場と仲間 → Day2 想い → Day3 ミニ解説（うっ血の見分け1点）→ Day5 心不全勉強会（月2）の予告 → Day6 声 → Day7 入会案内。
- 最終ゴール：有料会員化。
- 他ルート導線：デバイス（CRT/補助循環）、循環器（基礎）、AI（重症化予測など）。

## 5. AI活用ルート（ai）＝7日教育型（やや情報型）
- 目的：看護×AIの実践（業務効率・学習・情報整理）。
- 対象：AIに興味、学習や記録を効率化したいナース。
- Day0 歓迎 → Day1 仲間の活用例 → Day2 なぜ今AIか → Day3 すぐ使えるTips1つ → Day5 AI勉強会（月2）の予告 → Day6 変化の声 → Day7 入会案内＋（コミュニティでのAI企画）。
- 最終ゴール：有料会員化／AI勉強会参加。
- 他ルート導線：全ルート（AIは横断テーマ）、イベント。

## 6. 文房具・便利グッズルート（goods）＝情報フィード型（軽量）
- 目的：現場で使える道具情報の提供。間口を広げる入口（学習前段）。
- 対象：勉強より先に「便利グッズ」目的で登録した層。
- Day0 歓迎＋人気グッズ紹介 → Day2 用途別おすすめ（各社サイト/販売情報）→ Day4 ナースまつり物販・新商品 → （以後）不定期の新商品・販売情報フィード。
- 最終ゴール：一次＝商品情報の継続購読／イベント来場。二次＝学習ルート・会員へ送客。
- 他ルート導線：イベント（物販＝ナースまつり）、心電図/循環器（「学びも軽くどうぞ」）。
- 注意：教育色は薄く。押し売りしない。アフィリエイト/各社リンクは tracked-link で計測。

## 7. イベント情報ルート（event）＝情報フィード型（軽量）
- 目的：ナースまつり・セミナーの案内と集客。
- 対象：イベント目的の登録者、全ルート横断の再接触。
- Day0 歓迎＋直近イベント案内 → 申込導線（フォーム/LIFF）→ 開催前リマインド → 当日案内 → 終了後フォロー（アーカイブ/次回）。
- 最終ゴール：イベント参加。二次＝学習ルート・会員へ送客。
- 他ルート導線：全ルート（イベントは全員に効く）、文房具（物販）。

> **ルート横断の原則**：各ルートDay7後・イベント後に「Hubで興味を追加」を案内 → `I:` タグが増える → 横断セグメント配信が強くなる。

# ④ CRM全体マップ（一人の成長）

```
                         ┌──────────────────────────────────────────┐
                         │              be Navigator CRM             │
                         └──────────────────────────────────────────┘

 [獲得]        [入口]            [育成]              [接点]         [転換]        [継続]
 LINE登録  →  Hub（目的選択） →  ルート(7)        →  イベント    →  会員(¥2000) → 継続/コミュニティ
   │            │                 │                  │              │              │
 friend_add   ?page=hub        tag_added で        E:タグ        M:有料会員     S:アクティブ
 でhub-entry  で1つ選択         ルートシナリオ      で集客/後追い  B:入会         勉強会/Discord
 シナリオ起動  →I:＋R:タグ付与   Day0..Day7配信     来場=参加記録  決済連携       週5回/236本

   状態タグの流れ：
   S:オンボ中 ──(完走)──▶ S:オンボ完了 ──(反応継続)──▶ S:アクティブ
        │                                   │
        └─(無反応)─▶ S:離脱/休眠 ◀───────────┘ ──(再接触)──▶ 再エンゲージ配信

   会員の流れ：
   M:無料 ──(入会)──▶ M:有料会員 ──(解約)──▶ M:休会/退会 ──(声かけ)──▶ M:再入会候補 ──▶ M:有料会員

   横断：どのルートにいても イベント告知・興味追加(Hub再訪) で I: が増え、
        セグメント配信(R/I/M/B/S)が一人ひとりに最適化されていく。
```

**読み方**：登録者は「獲得→入口→育成→接点→転換→継続」を進む。各段階で**タグが状態を更新**し、**シナリオが次の一手を自動配信**する。会員化は終点でなく、コミュニティでの継続学習が本当のゴール。

# ⑤ データ設計（タグ／metadata／シナリオの役割）

## 3層の責務分離
| 層 | 役割 | 持つもの | 持たないもの | 参照のされ方 |
|---|---|---|---|---|
| **タグ** | 状態・集合所属（真偽） | ルート/興味/会員/イベント/行動/診断/ステータス | 連続値・長文・履歴明細 | tag_added起動・broadcastセグメント・シナリオ条件(tag_exists)・**管理画面の絞り込み** |
| **metadata** | 値・属性・個別化 | diagnosis_type/scores、interests配列、最終イベント、購入参照、表示用属性 | 「集合で配信したい状態」（それはタグ） | `{{metadata.X}}` 個別化・metadata_equals条件・分析 |
| **シナリオ** | 動き（自動配信の連なり） | ステップ文面・配信間隔・条件分岐 | 顧客状態そのもの | trigger（friend_add/tag_added/manual）で起動 |

## 設計指針
- **「集合で配信／分岐したい」=タグ**。「一人を個別化／記録したい」=metadata。迷ったらタグ（クエリしやすい）。
- metadata は JSON（D1の friends.metadata）。**metadata値での大量スキャンはしない**＝高頻度の絞り込みはタグに昇格させる。
- 履歴明細（購入の各明細、参加した全イベント）は将来は専用テーブル化を検討。当面はタグ（達成フラグ）＋metadata（最終値）で軽く持つ。

## 10,000人規模の運用設計
- **配信スループット**：cron 5分毎・1回最大40通（CF無料枠のサブリクエスト制限）。10,000人に一斉Day0は分散必須 → broadcast はバッチ＆stealth jitter、シナリオは相対配信で自然に分散。**大量同時enrollを避ける**（単一選択採用＝R6回避もこの一環）。
- **タグ設計**：カテゴリ制で総数を抑制（〜35＋イベント増分）。friend_tags のインデックス前提で絞り込み高速。
- **metadata肥大防止**：確定キーのみ（diagnosis_5項目＋interests＋数個）。際限なく増やさない。
- **冪等性**：enrollFriendInScenario は既enrollでno-op、タグは重複排除。再入・二重操作に強い。
- **将来**：購入/参加など増える明細は専用テーブル、スコアリング/LTVは別カラム or 集計ジョブへ（V3.0）。

# ⑨ 運営目線：管理画面で「一目で分かる」CRM像

## 友だち詳細（1画面で見える理想像）
ある登録者を開くと、以下が即座に分かる：

```
┌ 田中 花子（友だちID / LINE表示名 / 登録日）  [M:有料会員] ───────┐
│ 興味     : I:心電図, I:循環器, I:イベント                          │
│ 現在のルート: 心電図ルート（S:オンボ完了） → 循環器ルート（S:オンボ中）│
│ 診断     : D:L（語れるようになりたいタイプ） / scores F1 P0 V0 L11  │
│ 会員     : M:有料会員（入会 2026-05-01・継続中）                   │
│ 行動     : B:診断完了, B:アプリ登録, B:勉強会参加×4, B:グッズ購入   │
│ イベント : E:ナースまつり2026（来場）                              │
│ 配信予定 : 循環器Day3（明日）／ 会員向け週次                        │
│ 直近の配信/反応 : Day2開封・リンククリック（tracked-link）          │
└──────────────────────────────────────────────┘
```

## どの問いに、どのデータで答えるか
| 運営の問い | データ源 |
|---|---|
| 何に興味があるか | `I:` タグ（複数）＋ metadata.interests |
| 何を購入したか | `B:グッズ購入` ＋（将来）購入明細テーブル／metadata |
| 何を受講したか | `B:勉強会参加`/`B:アーカイブ視聴` ＋ S: 進捗 |
| どのイベントへ来たか | `E:` タグ（イベント単位） |
| 会員かどうか | `M:` タグ（1つ） |
| どんな配信を受けるか | 現在のルートシナリオ＋ライフサイクル＋セグメント条件（R/I/M/S） |
| 今どの段階か | `S:` タグ（オンボ中/完了/アクティブ/離脱/休眠） |

## 運営アクション（タグ起点で完結）
- 「`I:循環器` かつ `M:無料` の人へ循環器シリーズ紹介」→ broadcast セグメント配信。
- 「`B:入会ページ閲覧` かつ not `M:有料会員`」→ 後追いシナリオ。
- 「`E:ナースまつり2026` 来場者」→ お礼＋次回案内（event-followup）。
- 「`S:休眠`」→ 再エンゲージ（reengagement）。

→ **運営は「タグで母集団を作る→シナリオ/配信を当てる」だけ**。個別作業を増やさず10,000人を回す。

## 付録：実装の前提（事実確認済・コード参照）
- LIFF：単一アプリ `2010453320-O9UsF9z4`、`?page=` 分岐（main.ts）。Hubは `?page=hub` を加算。
- `/api/liff/*` は認証自動スキップ＝公開。route-select は diagnosis endpoint がお手本。
- tag_added：`scenarios.trigger_tag_id` 一致で自動enroll（friends.ts:436 / services/friend-tag-attach.ts）。**生 addTag ではなく発火経路を使う**（リスクR1）。
- friend_add：follow時に該当シナリオを複数enroll＋即時配信（liff.ts:847-881）。
- 配信：cron 5分毎・最大40通／回・stealth jitter（step-delivery.ts）。`expandVariables` で `{{metadata.X}}`/`{{#if_metadata.X}}`/`{{type_tip:dayN}}` 展開。
- broadcast：タグ／metadata_equals でセグメント配信可。
- D1スキーマ変更：本設計の範囲では**不要**（tags/friend_tags/scenarios/scenario_steps/friends.metadata で成立）。
- 関連：`docs/benavi-onboarding-scenario.md`（心電図ルート文面正本）、`beナビ_CRM_入口分岐_設計レビュー_v0.md`（入口設計v1＋Hubワイヤーフレーム）。
