> goods（文房具・便利グッズ）/ event（イベント情報）の「情報フィード型（軽量）」ルート設計。
> 教育型7日テンプレ（[benavi-onboarding-template.md](benavi-onboarding-template.md)）とは構造が異なるため別テンプレとして定義。
> **現段階は設計案＋本文案**：固定UUID発番・SQL生成・D1適用・deploy・broadcast実配信は未実施。

## 0. 結論：2層モデル

フィード型は「固定N日シナリオ」では表現できない（不定期・日付固定のため）。そこで**2層**に分ける：

| 層 | 役割 | 実装 |
|---|---|---|
| **層1：導入ミニシナリオ** | 登録直後の歓迎＋方向づけ＋一次導線（短い） | `scenarios`（tag_added・relative・2〜3ステップ） |
| **層2：以後のフィード** | goods=新商品/販売情報、event=各イベント案内・リマインド | **R:タグ宛 broadcast**（`broadcasts` target=tag）。シナリオ化しない |

- event の「開催前リマインド／当日／終了後フォロー」は**イベントごとに scheduled broadcast** を都度作る（イベント日が毎回変わるため）。
- **エンジン改修は不要**：`broadcasts` の tag 配信・`scheduled_at` を使うだけ。

## 1. なぜこの形か（教育型との違い）

- 教育型＝「読める→使える→語れる」の固定アーク（7日）。フィード型＝**間口を広げる入口**で、一次ゴールが「情報購読／イベント来場」、二次が「学習ルート・会員へ送客」。
- goods の「以後不定期の新商品フィード」は終わりが無い → シナリオ（有限ステップ）に不向き。broadcast が適合。
- event は日付固定（リマインドは“開催3日前”等）→ 相対delayでは出せない。scheduled broadcast の `scheduled_at` が適合。

## 2. 共通設定・整合性

- 導入ミニシナリオ：`trigger_type=tag_added` / `trigger_tag_id=R:<route>` / `delivery_mode=relative` / 診断分岐なし。
- **T1（intro Push 抑制）との整合**：route-select は R:タグに active な tag_added シナリオがあると intro Push を抑制する（[liff.ts](../apps/worker/src/routes/liff.ts) `hasActiveTagAddedScenario`）。→ 導入ミニシナリオを置けば **Day0 が歓迎を兼ね、二重 Push にならない**（教育型と同じ挙動）。
- **⚠️ 配信禁止帯 JST 21:00〜翌8:00**：
  - 層1シナリオは `clampToDeliveryWindow()` で自動ガード（教育型と同じ）。
  - **層2 broadcast は `scheduled_at` を手動設定するため自動ガードが効かない。運用ルールとして禁止帯に置かない**（§5 で明記）。
- メッセージ型：`text` 中心。商品/イベントは画像付き `flex`/`image` も可（broadcast 側）。

## 5. 層2：フィード運用 Playbook（broadcast）

> 層2は**シナリオではなく broadcast**。管理画面 or 運用で都度配信する。

### 配信トーンの大原則：**価値提供8：告知2**
フィード型はどうしても宣伝色が強くなりやすい。be Navigator らしい配信トーンを保つため、**「価値提供8：告知2」を基本**とする（運用担当が増えても守るルール）。

- 商品紹介だけで終わらせず、**「どう使うと便利か」**を必ず添える。
- イベント募集だけで終わらせず、**「参加すると何が学べるか／何が得られるか」**を必ず添える。
- **「買ってください」「参加してください」で終わらせない**——役に立つ情報が主、案内は従。
- 迷ったら「これは読んだ人の役に立つか？」を先に問う。告知だけの回を続けない。

### goods フィード（R:文房具 宛）
- 内容：新商品・販売情報・季節物・まつり物販。`flex`/画像で見せると効果的。
- 頻度の目安：**月数回まで**（間口広げ目的・押し売り回避）。教育色は出さない。
- 計測：各社/アフィリンクは **tracked-link**。購入確認できた人は `B:グッズ購入` タグ。
- 送客：たまに「学びも軽くどうぞ」を1行（しつこくしない）。

### event フィード（イベントごとに scheduled broadcast）
イベント1件につき、以下を**scheduled broadcast でセット**（`scheduled_at` を各日時に）：
1. 告知（決定時）
2. 開催前リマインド（例：3日前・前日 or 当日朝）
3. 当日案内（開始前）
4. 終了後フォロー（アーカイブ／次回案内）
- 対象：基本 `R:イベント` 宛。来場者は `E:<イベント名>` タグ → 終了後フォロー（`event-followup`）で絞り込み。
- 申込：フォーム／LIFF。参加実績は `B:勉強会参加` 等の行動タグへ。

### 共通禁則・ガード（必ず守る）
- ⚠️ **配信禁止帯 JST 21:00〜翌8:00 に broadcast を置かない**（`scheduled_at` を手動で禁止帯にしない）。
- 二重送信・誤アカウント送信を避ける（dedup・対象アカウント確認）。
- 秘密情報（友だちID・トークン・顧客データ）を本文/リンクに混ぜない。
- 押し売りしない・教育色は薄く（フィード型の原則）。

## 6. タグ運用（フィード型）

- ルート：`R:文房具` / `R:イベント`（tag_added 起動）、`I:文房具` / `I:イベント`（恒久興味）。
- 行動：`B:グッズ購入` / `B:勉強会参加` / `E:<イベント名>`（来場者）。
- 送客先：学習ルート（Hub 再訪 → ecg/circ/device/hf/ai）・有料会員。
