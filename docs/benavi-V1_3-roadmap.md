## 0. 前提と位置づけ

| 項目 | 状態 |
|---|---|
| V1.2 | ✅ 完成（7ルート本番稼働・CRM基盤完成） |
| V1.2.1 運用復旧 | ✅ 完了（LINE公式の応答方法を **手動チャット＋応答メッセージ** へ復旧。「今週の予定」「お問い合わせ」正常動作） |
| Worker × LINE公式 共存 | ✅ 確認済み（二重返信なし。Worker auto_reply=「テスト」1件のみ／automations 0件） |

**V1.3 ＝ Activation フェーズ。**
- 目的は **新機能の追加ではなく、V1.2で完成したCRMを実際に運用し、実データから学ぶこと**。
- **到達目標 ＝「LINE友だちを CRM ユーザーへ転換する」**。676人全員を D1 に登録することではなく、**興味を持った人を自然に CRM へ参加させる**こと（"ギャップを埋める"発想は採らない）。
- **D1 friends ＝「CRM参加者」**。これは「Workerが認識している人」ではなく、**興味選択を経て CRM に参加したユーザー**を意味する。
- **コード変更は最小限**。運用・計測・改善を最優先する。
- 設計思想は本書で**確定**するが、**実装変更は実データ → Decision Gate を経てから**行う。

## 1. 設計思想（確定）

### 1.1 Hub ＝「興味を育てる画面」

- Hub は **「ルートを決める画面」ではない**。**「興味を育てる画面」** である。
- 興味は固定の属性ではなく、**変化していくもの**として扱う。
- Hub は **Preference Center（興味管理）の入口** であると同時に、**CRM Entry Point（CRMへの参加点）** として位置づける。興味を1つ選んだ瞬間が「CRM参加」＝ LINE友だちが CRMユーザーへ転換する地点。

### 1.2 ユーザー向け語彙

| 区別 | 表現 |
|---|---|
| 内部名称（開発・コード・docs） | **Hub**（`?page=hub` / `client/hub.ts`） |
| ユーザー向け | **「興味を選ぶ」**（初回）／**「興味を追加する」「興味を変更する」**（V1.4以降） |

- **「Hub」「ルート」という語はユーザーに表示しない。**

### 1.3 興味の段差設計

| 版 | Hub の役割 | 選択 | 実装 |
|---|---|---|---|
| **V1.3** | 今、一番興味があるテーマを知る | **単一選択** | 最初の興味を1つ取得。複数選択は実装しない |
| **V1.4** | 興味の追加・変更 | **複数選択を解禁**（複数 R: タグ保持を許可・**優先順位なし**） | 「興味は変化する」前提の追加・変更導線 |
| **V2** | 興味の管理 | — | 興味管理画面（表示／追加／削除／変更） |

### 1.4 Activation ファネル（LINE友だち → CRMユーザーへの転換）

V1.3 の主軸は、このファネルを各段で観測し、転換を増やすこと。

```
LINE友だち（676・LINE OA母集団）
   ↓  Broadcast 等でHub提示
Hub到達（Preference Center / CRM Entry Point を開く）
   ↓
興味選択（R: 付与）
   ↓
CRM登録（D1 friends ＝ CRM参加者）   ← ここで「LINE友だち→CRMユーザー」転換が成立
   ↓
Day0 開始（オンボーディング配信スタート）
   ↓
Day1 以降（育成シナリオ進行）
   ↓
イベント参加・学習継続（E: 等・継続関係）
```

- **「CRM参加」の定義 ＝ 興味を1つ選び D1 friends として登録された状態**。
- 各段の「通過数」と「段間の転換率」を KPI とする（§5 P0.5）。

## 2. タグ運用 標準仕様（確定）

### 2.1 基本思想

> **タグ＝「現在の状態」ではなく「ユーザーとの関係性・履歴」。**
> **原則、削除しない**（積み上げていく）。

- 興味タグは **追加のみ**。削除・変更は **V2** で検討。

### 2.2 タグ体系

| 接頭辞 | 意味 | 付与者 | 付与契機 | V1の削除/変更 |
|---|---|---|---|---|
| **M:** | 会員状態（Member） | 管理者（将来＝決済連携） | 入会／退会 | 管理者がライフサイクル更新（興味の不削除ルールとは別軸） |
| **R:** | 興味タグ（Hubで選択） | Worker 自動 | Hub選択（V1.3＝1つ／V1.4＝複数可・優先順位なし） | しない（追加のみ）。変更は V2 |
| **S:** | シナリオ進捗（Scenario） | Worker 自動 | シナリオ進行に応じて | 自動管理 |
| **E:** | イベント参加（Event） | Worker 自動 | イベント参加時 | しない（履歴として残す） |

> 補足：`D:`（診断結果 F/P/V/L）等の補助カテゴリも「別軸」原則の下で併存（興味・会員と混ぜない）。

### 2.3 別軸の原則

- **会員状態（M:）と興味（R:）は完全に別軸**として管理する。
- **「会員だから全興味タグを付与する」運用は禁止。** メンバーも一般ユーザーと同様、Hub で「今、一番興味があるもの」1つから開始する。

### 2.4 Broadcast 配信ルール

- **OR 条件を標準**とする（例：`R:イベント` **または** `R:AI`）。
- **AND 条件は V1 では使用しない。**

---

## 3. タグ実装方針（V1.3）

V1.3 では、設計思想は確定するが **実装は変更しない**。

- **変更しないもの**：`route-select` / タグ付与ロジック / D1 / Worker。
- **現行実装のまま運用**する（route-select は現状どおり `I:` + `R:` を同時付与。`R:` をセグメント・KPI の正とする。休眠する `I:` タグは無害として放置。整理は Decision Gate 後）。
- 実データ取得後、**Decision Gate** を設け、タグ体系・route-select 変更の必要性を判断する。

> つまり：**思想は今回確定／実装変更は行わない（データを見てから）。**

## 5. Issue 分割案

### P0.5｜運用KPI・ダッシュボード設計
- **P05-1** KPI定義書（指標・定義・目標値）［doc］
- **P05-2** データ取得元マッピング（D1タグ / messages_log / friend_scenarios / LINE分析）［data］
- **P05-3** ダッシュボード方式選定（admin拡張 / 定期D1クエリ / スプレッドシート）［decision］
- **P05-4** ベースライン計測（V1.3開始時点スナップショット：LINE友だち **676** / **CRM参加者 6**（うちテスト2・実ユーザーのHub利用 0）/ 各R:タグ現状値）［data］
  - **主KPI ＝ Activation ファネル**（§1.4）：LINE友だち → Hub到達 → 興味選択 → **CRM登録（＝CRM参加者数）** → Day0開始 → Day1以降 → イベント参加・学習継続。各段の通過数・段間転換率。
  - **CRM参加者数**（旧称 "D1 friends数"）＝ファネルの中核KPI。その推移を追う。
  - 補助KPI：興味分布（R:タグ／履歴の累積・ランキングなし）・シナリオ進捗（Day到達率）・キーワード応答発火数・LINE側友だち純増/ブロック（LINE分析参照）・転換（checkout＝Stripe側参照）。

### P0｜Hub導線公開
- **P0-1** Hub事前実機確認（676配信前に単体テスト：選択→タグ付与→intro抑制）［test］
- **P0-2** 既存友だち向けBroadcast原稿（「今、一番気になるテーマを1つ」・価値8:告知2）［content］
- **P0-3** 配信設計（セグメント／時刻＝禁止帯回避／**ライトプラン無料通数の上限確認**／配信ツール）［ops/decision］
- **P0-4** Broadcast配信実行（オーナー）［ops］
- **P0-5**〔要判断〕新規友だち導線：あいさつメッセージにHubを追加するか／P3まで待つか［ops/content］
- **P0-7** メンバー移行ルール：メンバーにも同じHubで1つ選んでもらう。**全興味タグ一括付与は禁止**（M:は維持）［ops］
- **P0-10**〔要確認〕Hub UIコピー：「ルート」「Hub」等の内部語がユーザーに露出している場合のみ最小修正。なければDecision Gateまで据え置き［code/content］

### P0-9｜タグ体系レビュー（Review Issue・コード変更なし）
- 現行実装を運用して得た実データを基に、route-select / タグ体系の変更要否を **Decision Gate** で判断するためのレビュー枠。V1.3では実装を変更しない。

### P1｜Broadcast運用開始（ECG♡PASS / Event）
- **P1-1** Broadcast運用ルール確定（価値8:告知2・禁止帯手動ガード・頻度・署名 be Navigator）［doc］

- **P1-4** セグメント配信（**OR条件**：`R:イベント` / `R:文房具` 宛）［ops］

### P2｜Tracked Link導入
- **P2-1** tracked-link現状調査（auto-track.ts挙動・liff.line.me SKIP既知）［code/investigation］
- **P2-2** 計測対象リンク棚卸し（ECG♡PASS 4社 / HP / checkout / GAS / forms / Canva）［doc］

## 6. 残る要判断

1. **新規友だち導線（P0-5）**：あいさつメッセージにHubを足すか／P3のメニューまで待つか
2. **P2（Tracked Link）を P1 より前倒し**するか（推奨：ECG♡PASS本番告知を最初から計測付きに）
3. **ダッシュボード方式（P05-3）**：admin拡張／軽量D1クエリ＋スプレッドシート
4. **配信ツール＆通数**：LINEマネージャーbroadcast or Workerbroadcast／ライトプラン無料通数上限

> 解決済：リッチメニューのエンドステート＝Hubを「興味を選ぶ・変える」常設入口として共存（既存の情報導線と役割を分離）。

## 8. 用語

| 語 | 意味 |
|---|---|
| Activation | 作ったCRMを実運用し、実データから学ぶフェーズ。到達目標＝LINE友だちをCRMユーザーへ転換 |
| Hub（内部名） | 興味を育てる画面。ユーザーには「興味を選ぶ／追加する／変更する」と表示 |
| Preference Center | 興味管理。Hubはその入口 |
| CRM Entry Point | CRMへの参加点。興味を1つ選んだ瞬間がCRM参加＝Hubが担う |
| CRM参加者（数） | 興味選択を経て D1 friends として登録されたユーザー（旧称 "D1 friends数"）。Activationファネルの中核KPI |
| Activation ファネル | LINE友だち→Hub到達→興味選択→CRM登録→Day0→Day1以降→イベント参加・学習継続（§1.4） |
| Decision Gate | 実データを基に、実装変更（route-select/タグ体系/複数選択）の要否を判断する関門 |
| R: / M: / S: / E: | 興味 / 会員状態 / シナリオ進捗 / イベント参加（§2.2） |
